JPA Issues + Advantages

N+1 Problem

Vendor → Products → Stock

Entity: Vendor

@Entity
class Vendor {
  @Id @GeneratedValue
  Long id;
  String name;

  @OneToMany(mappedBy = "vendor")
  List<Product> products = new ArrayList<>();
}

Vendor knows its products (one-to-many).

Entity: Product

@Entity
class Product {
  @Id @GeneratedValue
  Long id;
  String name;
  String category;

  @ManyToOne
  Vendor vendor;

  @OneToOne(mappedBy = "product")
  Stock stock;
}

Product points back to Vendor and has Stock.

Entity: Stock

@Entity
class Stock {
  @Id @GeneratedValue
  Long id;

  @OneToOne
  Product product;

  int available;
}

Stock tracks quantity for a single product.

Story: “All Products by Vendor”

  • The UI needs products grouped by vendor.
  • Developer calls `vendorRepository.findAll()`.
  • `@OneToMany` on `Vendor.products` is LAZY by default.
  • Serializer walks each vendor and calls `vendor.getProducts()`.
  • Each call triggers another SQL query.
// Step 1: controller/service
List<Vendor> vendors = vendorRepository.findAll();

// Step 2: lazy collection (default)
@OneToMany(mappedBy = "vendor", fetch = FetchType.LAZY)
List<Product> products;

// Step 3: serialization or view rendering
for (Vendor v : vendors) {
  List<Product> ps = v.getProducts(); // triggers SQL
  response.add(v.getName(), ps);
}

The code looks clean, but the database sees a storm.

What the DB Sees (N+1)

-- 1 query
SELECT * FROM vendors;

-- N queries (one per vendor)
SELECT * FROM products WHERE vendor_id = 1;
SELECT * FROM products WHERE vendor_id = 2;
SELECT * FROM products WHERE vendor_id = 3;
...

One request turns into many. That’s the N+1 problem.

Why Bidirectional Helps

  • Case 1: Show all products for a vendor (vendor → products).
  • Case 2: Show products in a category with vendor details (product → vendor).

Same graph, different entry points.
